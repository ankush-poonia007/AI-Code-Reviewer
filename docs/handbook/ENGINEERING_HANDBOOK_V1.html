🎯 3. Vision & Goals

General objective layout defining what the software project achieves and its core quality boundaries.

Product Purpose
Provide instant, automated code auditing using LLM orchestrators without cloud vendor lock-in. Supports locally hosted models and cloud model routes (e.g. Gemini, Groq, Ollama).
Core Goals
• Provider Agnosticism: Swap LLM engines dynamically by config changes.
• Atomic Persistence: Stateless pipeline execution committed to local relational tables.
• High-Fidelity Telemetry: Record token metrics (prompt, completion) and processing speeds.

⚖️ 4. Engineering Rules

The project's constitution. The following rules govern database access, modeling boundary layouts, and dependency routing rules.

✓ Allowed Practices
  • Querying tables strictly inside Repository files.
  • Mapping Pydantic request models to SQLAlchemy DB models inside Service Layer boundaries.
  • Verifying provider credentials and system environment files at application boot time.
  • Mocking third-party network APIs during testing cycles.
✗ Strictly Prohibited
  • Direct DB queries or model imports inside API routers.
  • Exposing raw SQLAlchemy models outside the controller layer (always return Pydantic schemas).
  • Hardcoding provider configurations inside the source code (always consume settings class values).
  • Bypassing unit testing pipelines for backend integrations.

⚙️ 5. Design Philosophy

Our architectural design patterns follow production coding methodologies to ensure structural longevity.

Separation of Concerns (SoC)
Keep HTTP route adapters, business workflow services, repository data layers, and database ORM entities strictly decoupled.
Stateless Services Pattern
Service layers retain no state variables. This ensures thread-safety under asynchronous worker conditions (e.g. Uvicorn).
Dependency Injection (DI)
Pass database sessions, providers adapters, and prompt templates to class constructors dynamically to ease test fixture mockup creation.

🧩 7. Layer Duties & Boundaries

A matrix detailing exact duties across components of the backend codebase.

API Layer (FastAPI Route Handlers)
• Intercepts incoming requests.
• Validates parameters, schemas, and binary file streams inputs.
• Catches system errors and formats HTTP Exception returns (ADR-039).
Service Layer (Workflow Orchestrators)
• Executes core transactional logic loops.
• Assembles prompts using parameters.
• Calls provider adapters.
• Parses responses and saves schema data models to Repositories.
Repository Layer (Data Access Abstraction)
• Formulates SQLAlchemy query executions.
• Manages transactional entities lifecycle mapping operations (insert, update, delete).
Domain Database Tier (SQLAlchemy Entities)
• Declares SQLAlchemy structural maps.
• Enforces unique indexing, relationships validation, and auditable timestamps metadata.

📝 9. Coding Standards

Rules governing the creation of Python modules, error handling layers, and logging.

Type Safety Policy
Every method declaration must contain type annotations. Use Python 3.11 type union styles (e.g. dict[str, str] | None) instead of importing from legacy typing modules.
Centralized Logging System
All modules consume loguru using labels (e.g., logger.bind(component="review_service")). Standard print outputs or logging modules are banned.
Configuration Policy
Read configurations exclusively from backend/config.py. API credential keys must be checked on startup; fail-fast if settings validation checks fail.

💾 14. Database Responsibilities

The persistence layout targets SQLite for local simplicity and zero footprint configurations.

SQLite WAL Mode (Write-Ahead Logging)
Enabled on engine connect. Prevents concurrent dashboard read actions from locking during ongoing AI review write queries.
Cascade Deletions Enforcement
Parent reviews rows automatically trigger cascade deletions on downstream issues and review_executions telemetry records to prevent database clutter.
UUID primary identities generation
Models inherit UUIDMixin generating 36-character random UUID string keys, preventing sequence scan issues.

🧠 15. AI Pipeline Execution Steps

Workflow detailing how source files are audited by Large Language Models.

1. Code Ingestion & Language Check
Verify file payload is under the 5MB setting limit. Match file extension to resolved language enums.
2. Prompt Rendering (ADR-030)
Combine system prompt blueprints (score scales, JSON rules) with dynamic parameters (untrusted code content blocks).
3. Provider Dispatch
Pass prompt content down to provider factories (Gemini / Groq SDK wrapper clients) using configurable timeouts (120s default).
4. Response Parsing (ADR-027)
Regex boundary extraction isolates structural JSON templates from returned text formats. Malformed blocks raise parsing exceptions.
5. Persistence Step
Map parsed keys directly to database classes, write to repositories inside transaction loops, and perform database session commits.

📑 23. Detailed Architecture Decisions Log

Detailed reasoning logs documenting key architectural decisions.

ADR-001: Strict Layered Architecture
Accepted

Decision: Divide backend code into clean modules (UI → API routes → Services → Databases Repositories → SQLite).

Reasoning: Enforces modularity, isolates business logic from SQL frameworks, and eases integration testing.

Scope: System Layout Approved: July 2026
ADR-002: Centralized Settings via Pydantic
Accepted

Decision: Use pydantic-settings inside backend/config.py as the sole settings manager. Direct env-reads outside the config file are banned.

Reasoning: Ensures runtime variables are checked and typed on startup, avoiding downstream KeyError issues.

Scope: Configuration Approved: July 2026
ADR-015: Shared BaseModel Base Class
Accepted

Decision: Create an abstract BaseModel parent class that houses standard system columns (id, created_at, updated_at).

Reasoning: Avoids duplicating database audit columns across model schemas, centralizing identity policies.

Scope: ORM Design Approved: July 2026
ADR-016: Decoupled ID & Audit Mixins
Accepted

Decision: Extract ID mapping features to UUIDMixin and auditing variables to TimestampMixin.

Reasoning: Improves code reuse. Models can inherit only the specific columns they require.

Scope: ORM Design Approved: July 2026
ADR-017: Centralized models/enums.py Domain Alignment
Accepted

Decision: Move severity levels, category types, languages, and pipeline status enums to a centralized models/enums.py module.

Reasoning: Eliminates circular database model imports and standardizes API serialization.

Scope: Domain Models Approved: July 2026

📚 27. Appendix: Glossary & References

Explanation of system acronyms and reference materials.

ADR (Architecture Decision Record)
Structured text document capturing an architectural decision, its context, reasoning, and system consequences.
WAL (Write-Ahead Logging)
SQLite configuration allowing read queries to execute concurrently while write queries update the database.
RAG (Retrieval-Augmented Generation)
System integrating external database queries with LLM prompting pipelines to supply localized file context.
